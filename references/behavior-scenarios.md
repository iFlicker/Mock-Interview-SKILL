# 端到端行为验收场景

本文件定义 Skill 的行为验收标准，不重复解释实现规则。机器可读用例位于 `tests/skill-contract-cases.json`，结构测试由 `scripts/test_skill_contract.py` 执行。对真实模型进行回归时，应把每个场景作为独立对话运行，并同时检查预期输出和禁止输出。

## 必须覆盖的场景

| 场景 ID | 用户行为 | 必须发生 | 禁止发生 |
|---|---|---|---|
| `resume_missing` | 未提供且找不到简历 | 原样要求提供简历并停止 | 通用题库面试、假装读到简历 |
| `multiple_resumes` | 找到多份可能简历 | 要求用户选择 | 擅自选择最新或文件名最像的一份 |
| `config_missing` | 有简历和岗位，但面试配置不足 | 展示相关默认选项并等待确认 | 在确认前提出面试题 |
| `skip_main_question` | 主问题后回复“跳过” | 当前主题不计覆盖，换自然问题 | 暴露主题编号或内部计数 |
| `pause_resume` | 暂停、讨论流程、继续 | 暂停期不记评分证据，恢复原主题、主问题和追问次数 | 重复开始条、清空进度 |
| `pressure_adjustment` | 中途调整压力值 | 限制到 1～100、记录变更；非核心主题不自动扩大追问预算 | 因压力升高自动增加问题、突破职业边界或 6 次追问上限 |
| `coach_evidence` | 教练模式下提示后重新回答 | 将后续证据标记为 `coached` | 把提示本身算作候选人证据 |
| `end_without_evidence` | 开始后立即结束 | 输出结束条；总分为证据不足；结论待定或不建议 | 输出数字 0 分或确定性通过结论 |
| `combined_interview` | “经理二面＋系统设计＋压力风格” | 正确拆分阶段、角色、形式和修饰，不叠加范围 | 把压力面试当成额外主题数量 |
| `director_depth_profile` | 总监面考察关键系统决策 | 使用 `deep` 和 4～5 次建议范围，重点转向决策、资源、风险和影响 | 只增加局部技术细节追问 |
| `follow_up_limit` | 同一主题持续产生可追问内容 | 主问题不计数；第 6 次追问后完成主题或结束 | 以矛盾未解决为由提出第 7 次追问 |
| `contradiction_triage` | 两项陈述看似不一致 | 先分类；缺上下文时中立澄清 | 直接指控造假或把已解决差异计为风险 |
| `report_consent` | 评价后拒绝或未明确同意报告 | 不创建文件 | 将沉默、模糊回应当成授权 |
| `evidence_gap_selection` | 已完成 3 次追问，当前机制充分但异常边界缺失 | 提出第 4 次追问，补异常或边界证据 | 仅因已经能够评分就结束，或重复询问充分机制 |
| `soft_minimum_completion` | 建议范围为 3～4，但 2 次追问后关键证据均已充分 | 允许提前完成并记录 `evidence_sufficient` | 为达到建议下限而凑问题 |
| `question_quality_gate` | 候选问题包含四个子问题或答案暗示 | 静默改写为单一、非诱导问题 | 展示检查过程或原样输出低质量问题 |
| `opening_closing_state` | 专业面进行开场和收尾 | 分别记录为 `opening`、`closing`；开场可产证据，收尾不评分 | 将两者计入主题或把收尾写成评分证据 |
| `blueprint_remains_frozen` | 面试中调整压力、重点或覆盖数量 | 只调整配置和提问计划，记录控制历史 | 修改评分维度或权重 |
| `report_consent_reopened` | 先拒绝报告，随后明确反悔 | 状态可恢复时重新进入报告授权并生成 | 补写无法恢复的内容 |
| `confirmed_scope_cap` | 达到用户确认的覆盖数量 | 完成适用收尾并结束 | 未经确认增加覆盖主题或考察项 |
| `partial_coverage_score` | 有部分维度可评分但覆盖率低 | 输出暂定数字总分、覆盖率限制，结论最高待定 | 把未知维度计为 0 分或给确定性通过结论 |

## 回归判定

- 场景通过必须同时满足全部 `expected` 和全部 `forbidden`。
- 文案不要求逐字一致，除非 SKILL 明确规定“原样输出”。
- 内部状态断言通过状态快照或结构化评审检查，不得要求正式面试阶段向用户展示内部状态。
- 新增控制指令、阶段或 payload 不兼容字段时，必须先增加或更新场景，再修改实现文档。
- 至少在修改 `SKILL.md`、`interview-policy.md`、`session-state.md` 或 `scoring-rubrics.md` 后运行契约测试。
