# 端到端行为验收场景

本文件定义 Skill 的行为验收标准，不重复解释实现规则。机器可读用例位于 `tests/skill-contract-cases.json`，结构测试由 `scripts/test_skill_contract.py` 执行。对真实模型进行回归时，应把每个场景作为独立对话运行，并同时检查预期输出和禁止输出。

## 必须覆盖的场景

| 场景 ID | 用户行为 | 必须发生 | 禁止发生 |
|---|---|---|---|
| `resume_missing` | 未提供且找不到简历 | 原样要求提供简历并停止 | 通用题库面试、假装读到简历 |
| `multiple_resumes` | 找到多份可能简历 | 要求用户选择 | 擅自选择最新或文件名最像的一份 |
| `config_missing` | 有简历和岗位，但面试配置不足 | 展示相关默认选项并等待确认 | 在确认前提出面试题 |
| `skip_main_question` | 主问题后回复“跳过” | 当前主题不计覆盖，换自然问题 | 暴露主题编号或内部计数 |
| `pause_resume` | 暂停、讨论流程、继续 | 暂停期不记评分证据，恢复原主题、主问题和追问次数 | 重复开始条、清空进度 |
| `pressure_adjustment` | 中途调整压力值 | 限制到 1～100、记录变更；非核心主题不自动扩大追问预算 | 因压力升高自动增加问题、突破职业边界或 6 次追问上限 |
| `immersive_feedback_boundary` | 正式面试中用户要求即时点评 | 说明结束后统一反馈，并继续自然提问 | 给出即时点评、参考答案或切换为实时辅导 |
| `end_without_evidence` | 开始后立即结束 | 输出结束条；总分为证据不足；结论待定或不建议 | 输出数字 0 分或确定性通过结论 |
| `combined_interview` | “经理二面＋系统设计＋压力风格” | 正确拆分阶段、角色、形式和修饰，不叠加范围 | 把压力面试当成额外主题数量 |
| `director_depth_profile` | 总监面考察关键业务或系统决策 | 使用 `deep` 和 4～5 次建议范围，重点转向决策、资源、风险和影响 | 只增加局部执行细节追问 |
| `follow_up_limit` | 同一主题持续产生可追问内容 | 主问题不计数；第 6 次追问后完成主题或结束 | 以矛盾未解决为由提出第 7 次追问 |
| `contradiction_triage` | “负责整体方案”与“关键方向由负责人确定”看似不一致 | 先分类；缺上下文时中立澄清 | 直接指控造假或把已解决差异计为风险 |
| `report_consent` | 评价后拒绝或未明确同意报告 | 不创建文件 | 将沉默、模糊回应当成授权 |
| `evidence_gap_selection` | 已完成 3 次追问，当前机制充分但异常边界缺失 | 提出第 4 次追问，补异常或边界证据 | 仅因已经能够评分就结束，或重复询问充分机制 |
| `soft_minimum_completion` | 建议范围为 3～4，但 2 次追问后关键证据均已充分 | 允许提前完成并记录 `evidence_sufficient` | 为达到建议下限而凑问题 |
| `question_quality_gate` | 候选问题包含四个子问题或答案暗示 | 静默改写为单一、非诱导问题 | 展示检查过程或原样输出低质量问题 |
| `opening_closing_state` | 专业面进行开场和收尾 | 分别记录为 `opening`、`closing`；开场可产证据，收尾不评分 | 将两者计入主题或把收尾写成评分证据 |
| `blueprint_remains_frozen` | 面试中调整压力、重点或覆盖数量 | 只调整配置和提问计划，记录控制历史 | 修改评分维度或权重 |
| `report_consent_reopened` | 先拒绝报告，随后明确反悔 | 状态可恢复时重新进入报告授权并生成 | 补写无法恢复的内容 |
| `confirmed_scope_cap` | 达到用户确认的覆盖数量 | 完成适用收尾并结束 | 未经确认增加覆盖主题或考察项 |
| `partial_coverage_score` | 有部分维度可评分但覆盖率低 | 输出暂定数字总分、覆盖率限制，结论最高待定 | 把未知维度计为 0 分或给确定性通过结论 |
| `natural_answer_bridge` | 回答中给出与下一问直接相关的具体事实 | 用简短承接准确引用该事实，再提出一个主要问题；无有意义内容时直接提问 | 机械重复“好的”、补写候选人未说过的结论、在承接句暗藏第二个问题 |
| `challenge_move_gradient` | 同一主题从背景不清逐步形成可验证方案 | 先澄清或具体化；背景建立后按证据缺口选择取舍、验证、边界或约束变化，并保持动作变化 | 基础事实未明时直接构造复杂反事实、连续轻微改写同一种挑战、因压力值自动升级 |
| `finite_interviewer_knowledge` | 材料只说明采用了某方案，未说明决策原因或内部限制 | 把材料当作待核实背景，中立询问原因；使用推断时明确校准并允许纠正 | 假定成本、团队意图、内部数据或真实结果，故意使用错误专业前提测试候选人 |
| `non_technical_role_defaults` | 用户以区域运营经理、品牌营销经理等非技术岗位进行综合专业面 | 根据目标岗位选择经营分析、专业方法、业务结果和协作等相关主题 | 把“专业面”默认解释成编程、算法或系统架构面试 |
| `interaction_channel_choice` | 展示完整配置且用户此前未指定交互方式 | 在同一轮请用户调整配置或直接选择按当前配置开始文字/语音面试 | 先确认配置再固定追加一次通道确认、选择前启动网页或提出正式问题 |
| `agent_text_no_web` | 用户选择文字面试 | 继续现有应用内逐轮问答 | 启动网页、MCP、本地服务或浏览器 |
| `web_voice_connection_gate` | 用户选择语音面试且配置已确认 | 先准备服务并提供 URL，再自动打开网页；网页实际连接后才输出一次开始边界 | 浏览器先打开而提示滞后、展示技术配置、把打开动作当作连接成功、提前提问 |
| `web_voice_boundary_display` | 语音面试开始或结束 | Agent App 显示长分界线，网页只显示一次“面试开始”或“面试结束” | 网页显示长横线、朗读边界、因状态事件和边界消息重复显示结束 |
| `web_voice_activity_feedback` | 网页已连接但首题未到达，或回答已由服务确认但下一问未到达 | 显示带旋转图标的临时准备或“思考中...”状态；回答气泡标记“已发送”；正式消息到达后清除 | 让用户面对空白页面、伪造进度、朗读临时状态或写入问题历史和评分证据 |
| `web_voice_first_start_retry` | 第一个本地服务子进程在健康检查前退出 | 同一次启动请求内自动退避重试并成功；临时诊断不泄露内容 | 首次瞬时失败就要求用户手动重试、保留启动日志或 token |
| `web_voice_wait_persistence` | 问题已推送，候选人仍在思考或编辑，连续多次短等待超时 | 同一 Agent 回合持续短轮询并保持当前问题；网页显示真实 Agent 心跳 | 输出最终答复、让任务进入 idle、关闭会话、评分或提出下一题 |
| `web_voice_single_waiter` | 持续等待工具返回运行中的 cell 或 shell session | 续读原运行会话；运行时拒绝第二个等待者，异常退出后可回收租约 | 为同一会话再次执行 wait、形成并发事件读取者 |
| `web_voice_rephrase_reply` | 候选人已回答，面试官需要重新表述或要求展开 | 用新问题 ID 推送 interviewer_question，网页解锁并接受新回答 | 用 interviewer_message 要求回答、继续等待旧问题 ID、永久显示已提交 |
| `web_voice_idempotency` | MCP 重试相同问题或网页重复提交回答 | 只产生一个气泡、一次自动朗读和一次证据消费 | 重复问题历史、重复回答或重复评分证据 |
| `web_voice_reconnect` | 网页断开、刷新并重新连接 | 通过 snapshot 和 cursor 恢复消息与回答，保持同一正式会话 | 重复开始边界、重复朗读历史、丢失或乱序事件 |
| `web_voice_switch_text` | 用户在网页或 Agent App 切回文字面试 | 保留配置、当前问题、历史和证据，在当前应用继续 | 重复开始边界、清空进度、继续向网页推题 |
| `web_voice_deploy_failure` | 当前 Agent 在远程环境或本地服务无法恢复 | 如实说明限制并允许立即切回文字面试 | 伪造 localhost 可访问、要求普通用户配置 MCP 或端口 |

## 回归判定

- 场景通过必须同时满足全部 `expected` 和全部 `forbidden`。
- 文案不要求逐字一致，除非 SKILL 明确规定“原样输出”。
- 内部状态断言通过状态快照或结构化评审检查，不得要求正式面试阶段向用户展示内部状态。
- 新增控制指令、阶段或 payload 不兼容字段时，必须先增加或更新场景，再修改实现文档。
- 至少在修改 `SKILL.md`、`interview-policy.md`、`session-state.md` 或 `scoring-rubrics.md` 后运行契约测试。
