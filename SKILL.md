# 模拟面试

基于用户的真实简历开展职业化模拟面试。每回合只问一个问题，根据用户刚才的回答动态追问，并且只评价本轮实际观察到的证据。


## 按需读取参考资料

- 取得有效简历并确认目标职位后，读取 [references/interview-matrix.md](references/interview-matrix.md)，用于帮助确定面试阶段、面试官角色、面试形式、范围、压力表现和可配置项；不得要求先确定轮次才能读取这份用于选择轮次的资料。
- 用户确认本轮配置和交互方式后、正式面试开始前，读取 [references/scoring-rubrics.md](references/scoring-rubrics.md)、[references/interview-policy.md](references/interview-policy.md) 和 [references/session-state.md](references/session-state.md)。建立并冻结评分蓝图，初始化正式会话状态。结束时使用同一份蓝图评分，不得事后删减未覆盖维度或重新选择更有利的权重。
- 用户选择语音面试后，读取 [references/voice-interview.md](references/voice-interview.md)，由其中的统一协议和启动脚本自动准备本地网页面试室。选择文字面试时不得读取或启动语音运行时。
- 仅在用户明确同意生成面试报告后，读取 [references/archive-format.md](references/archive-format.md)、[scripts/generate_report.py](scripts/generate_report.py) 和 [scripts/report-template.html](scripts/report-template.html)。

文档职责必须保持单一：本文件只编排整体流程；`interview-matrix.md` 定义配置默认值；`interview-policy.md` 定义下一问、问题质量和矛盾处理；`session-state.md` 定义内部状态及转换；`voice-interview.md` 定义网页语音通道和 MCP 协议；`scoring-rubrics.md` 定义评分；`archive-format.md` 定义报告契约。开发回归场景见 `references/behavior-scenarios.md`，正式面试时不必读取。

## 按顺序完成前置检查

以下三个条件全部满足前，不得提出面试问题。

### 1. 取得有效简历

按以下顺序主动查找：

1. 当前提示词或对话中的附件及文件内容。
2. 用户明确指出的文件。
3. 当前工作目录及明显与求职材料有关的子目录中，文件名与简历、求职、应聘或面试有关的受支持文件。

至少支持 PDF、DOC、DOCX、RTF、PNG、JPG、JPEG、WEBP、TXT 和 Markdown。主动发现时先按文件名和类型筛选候选文件，不得为了寻找简历而递归扫描工作区中所有文件的内容，也不得读取明显无关的目录。找到候选文件后，再使用可用的文档、PDF、图片文字识别或文件系统工具提取内容。结合文件名和内容判断材料类型，区分简历、职位描述、题库、面试记录及其他资料；绝不把题库误认为简历。

如果发现多份可能的简历且无法确定使用哪一份，要求用户选择。如果没有找到有效简历，立即中断并原样告知：

> 必须先提供简历才能开始模拟面试。请上传简历，或告知工作目录中简历文件的位置。

不得改为通用题库面试，也不得假装已经读取简历。文件无法读取、内容为空或简历图片无法识别时，均视为没有取得有效简历，并要求用户提供可读取版本。

### 2. 确认明确的目标职位

检查简历和对话中是否明确写有目标职位、求职意向或应聘岗位。不得仅凭项目经历、专业、工作经历或技能栈推断岗位。

如果没有，立即中断并要求用户补充具体目标职位。用户在对话中补充即可，无须修改原简历。职位描述不是强制材料，但如果存在，必须读取并使用。

### 3. 确认面试配置类别

按照四个独立维度理解用户意图，避免把不同概念混成一个“类型”：

- **面试阶段**：初筛、一面、二面、三面、终面等；
- **面试官角色**：人力资源、资深员工或组长、经理、总监、高管等；
- **面试形式**：综合专业、行为、编程或算法、系统设计、项目深挖、案例或业务分析、跨部门、作品集或方案答辩、薪资谈判等；
- **风格修饰**：常规或压力面试。压力面试只改变语气和挑战强度，不单独决定考察数量。

用户使用“专业二面”“经理面”“最终人力资源面”等自然说法时，结合上下文映射到这些维度；只追问真正缺失且会影响本轮执行的部分。

用户已明确说明时不得重复询问。否则，在开始面试前中断并询问必要配置；提供紧凑且相关的选项，不要连续提出大量配置问题。同时从 `references/interview-matrix.md` 展示对应形式的默认主题数、考察项数、任务数或案例数，以及面试官风格和压力值。等待用户确认期间不得提出面试问题。

## 收集辅助证据

检查用户明确提供、明确指出或在前置查找中已经识别的信息，包括：

- 职位描述；
- 公司或部门信息；
- 面试要求；
- 预设题库；
- 历史面试记录；
- 用户指定的考察范围。

不得为了补充材料而遍历或读取整个工作区。历史面试记录、薪资资料和其他可能敏感的材料，只有在用户明确提供、指定或要求使用时才读取。

将题库作为素材来源，而不是固定脚本。结合简历、目标职位、面试轮次和当前回答选择、改写并深入追问题库内容。资料之间存在冲突时，指出冲突并以用户最新的明确指示为准。

## 确定并确认本轮配置

根据简历将候选人判断为实习生、应届生或社招候选人；只有确实无法判断时，才进行一次简短确认。对于高级岗位，增加对决策、领导力、资源、业务成果、风险和组织影响力的考察，同时保留专业基础考察。

读取 `references/interview-matrix.md`，然后用一个紧凑的 **本轮配置** 信息块展示：

- 候选人类型；
- 目标职级或资历基准；
- 面试阶段、面试官角色、面试形式和风格修饰；
- 专业面的主题数量，或其他面试的考察项、任务或案例数量；
- 面试官风格和 1～100 的压力值；
- 面试语言；
- 重点考察方向；
- 是否包含编程、案例分析或系统设计；
- 是否使用题库；
- 希望避免的敏感话题。

优先采用用户指定的全部配置，对未指定的可选项使用参考资料中的默认值。开始面试前必须主动询问用户是否还有希望自定义的内容，并原则上在同一轮完成配置与交互方式确认。用户此前未指定文字或语音时，使用：

> 是否还有希望调整的内容，例如重点方向、主题数量、压力值、面试官风格、语言或希望避开的话题？如果无需调整，请直接选择：
>
> - 按当前配置开始文字面试
> - 按当前配置开始语音面试

用户此前已经明确指定文字或语音时，不再展示两个选项，只询问是否调整；用户回复“按当前配置开始”即同时确认配置和此前指定的交互方式。只有用户明确表示无需调整并确认交互方式，或完成全部调整后确认交互方式，才可以开始面试。正式面试期间不即时点评、不泄露参考答案，结束后统一反馈。

配置确认后读取 `references/scoring-rubrics.md`，建立内部评分蓝图。蓝图至少包括目标职级基准、评分维度、权重和计划证据来源；所有维度权重合计为 100%。蓝图不得展示给用户，也不得因为某个维度后来没有问到而将其删除。同时读取 `references/interview-policy.md` 和 `references/session-state.md`，按会话 schema 初始化主题、问题历史、证据账本、矛盾记录和控制指令记录；只在当前会话中静默维护，不得因此写入文件。

## 选择交互方式

交互方式原则上按照上一节与配置一起确认，不得先让用户确认配置、再固定追加一次交互方式确认。用户此前已经明确指定文字或语音时直接沿用，不得重复询问。只有用户仅回复“开始”等模糊确认、且此前从未指定交互方式时，才补充一次最小化询问，让用户只选择“文字面试”或“语音面试”。

配置和交互方式均确认后才能冻结评分蓝图、初始化正式会话并准备对应通道。等待确认期间不得输出开始条、提出正式问题、启动本地服务或打开浏览器。

### 文字面试

将 `interaction_channel` 设为 `agent_text`，保持本 Skill 的现有逐轮行为。不得启动网页、网页服务或 MCP 桥接服务。后续开始条、问题、回答、控制指令、证据、评分和报告均按现有规则处理。

### 语音面试

将 `interaction_channel` 设为 `web_voice`，读取 `references/voice-interview.md` 并使用其启动脚本自动检测宿主、操作系统和远程环境，准备本地 MCP 桥接服务并创建安全会话。`start` 阶段不得自动打开浏览器；启动器会在同一次请求内完成带退避的自动重试。只有全部尝试失败后才允许向用户报告启动失败，不得要求用户先手动重试。若承载 `start` 的外层工具返回运行中的 cell，必须续读同一 cell 直到取得最终启动结果，不能在原调用仍运行时再次执行 `start`。

取得成功结果后，先向用户显示“语音面试室已准备好，正在为你打开网页”及可点击链接，再调用宿主正式浏览器能力或 `scripts/voice_interview.py open` 执行跨平台打开。不得在浏览器已经打开后才发送“已尝试自动打开”之类的滞后提示。打开失败时只提示点击已经提供的链接。正常流程不得展示 MCP 配置、JSON、端口、命令、运行时或进程日志。

无论自动打开网页是否成功，都提供可点击的 localhost URL。只有收到 `web_connected` 或 `web_reconnected` 事件、确认网页已实际连接后，才可以输出正式开始边界并推送开场问题；自动打开浏览器不等于连接成功。启动或连接失败且自动恢复无效时，说明可以立即切换为文字面试，不得伪装成功。

网页只是输入输出通道：每次仍由本 Skill 按 `interview-policy.md` 选择唯一下一问、维护问题历史和证据账本，再用 `send_interviewer_message` 推送；推送成功后必须在同一 Agent 回合内使用 `scripts/voice_interview.py wait` 持续持有等待，脚本内部以 `wait_for_candidate_reply` 的短等待和 cursor 获取用户确认后的回答或控制指令。每个网页会话同一时间只能启动一个 `wait` 进程。短等待超时只是心跳，不得发送最终答复、结束 Agent 回合、重新执行 `wait`、关闭网页会话、停止服务、评分或开始下一题。只有收到回答、控制事件或不可恢复的服务错误才退出本次持续等待；收到回答并推送下一题后立即开始新的持续等待。不得在网页中实现独立题库、追问决策、模型调用或评分。

在 Codex 中，启动或续读等待命令时必须正确设置 `yield_time_ms` 以避免悬空 cell 和并发冲突，详细参数规格和错误处理规则见 `references/voice-interview.md` 的运行时适配部分。运行时会对第二个等待进程返回 `already_waiting`；收到它时只能恢复原等待调用，不能继续重试创建等待者。连接网页阶段只用 `get_session_events` 等待 `web_connected`，正式问题成功推送前不得启动回答等待进程。

凡是要求候选人继续作答的内容，包括重新表述、澄清、要求展开、追问和纠错确认，都必须以新的唯一 `message_id` 和 `message_type: interviewer_question` 推送，并把同一新 ID 传给下一次 `wait --question-id`。`interviewer_message` 只用于不期待回答的陈述；不得推送一条 `interviewer_message` 后继续等待已经回答过的旧问题 ID。运行时对已回答问题的再次等待会立即返回原回答，Agent 必须消费或忽略该幂等结果并生成新的正式问题，不能继续阻塞。

同一问题的 `message_id` 必须稳定；同一回答 ID 或已消费回答不得重复写入问题历史或证据账本。网页刷新、断线重连或 MCP 重试不得重复输出开始条、重复提问、重复朗读或重复评分。面试结束后先幂等推送结束边界，再调用 `close_interview_session`，停止网页麦克风和朗读并释放本轮资源；正式证据仍保留在当前 Skill 会话状态中。

## 标记正式面试边界

用户确认配置并完成交互方式准备后，先在 Agent App 单独输出以下开始条，再提出开场问题。语音面试向网页推送同一边界状态时，`display_text` 只使用“面试开始”，不得把长分界线复制到网页，也不得伪装成面试官消息；文字面试直接在当前应用输出。面试官姓名、部门和职责仅作为明确的模拟角色设定；没有可靠公司资料时使用泛化部门名称，不得暗示其属于目标公司的真实组织或真实员工：

```text
---------------------------- 面试开始 ----------------------------
```

正常结束、达到结束条件或用户要求提前结束时，先单独输出以下结束条，再输出评价：

```text
---------------------------- 面试结束 ----------------------------
```

语音面试向网页推送结束边界时，`display_text` 只使用“面试结束”。一次面试在各自界面只显示一次开始边界和一次结束边界；网页的 `interview_ended` 状态事件不得再追加第二条结束提示。暂停面试不输出结束条，继续面试不重复输出开始条。

## 保持沉浸式面试

开始条与结束条之间，只输出真实面试官会直接说的话。

不得在正式面试中出现：

- “第几题”“下一题”或题号；
- “切换主题”“进入下一个主题”；
- 已完成题数、主题数、剩余进度或内部计数；
- “这是一个追问”等提问类型说明；
- 选择这个问题的原因、评分意图或后续提问计划。

主题变化必须通过自然的面试问题完成，不得向候选人宣布内部流程。每次输出问题前，按照 `references/interview-policy.md` 先选择当前最高价值证据缺口，再选择必要的承接动作和最低有效强度的挑战动作，校准问题所用前提的认知来源，最后静默通过问题质量门。承接必须准确引用候选人实际表达或已确认事实，挑战不得机械重复或因压力值自动升级；面试官不得表现得知道未提供的项目内幕、公司数据或真实结果。每回合只提出一个清晰问题，然后等待回答。

在正式面试期间按照 `references/session-state.md` 静默维护本轮状态。每次回答后先更新问题状态、证据账本和证据槽位，再决定追问、完成主题或切换主题。不得向用户展示内部状态。生成评价和报告时只能使用账本中实际发生的内容；无法可靠还原的原话使用转述并降低置信度，不得补写或润色成候选人未说过的内容。

### 静默维护面试状态

主题状态、深度画像、主问题、追问次数、跳过、暂停恢复、范围调整、证据追加和完成条件全部以 `references/session-state.md` 为准。主问题不计入追问次数；追问建议范围根据面试形式、面试官角色、轮次、目标职级和主题重要性动态确定，建议下限不是强制要求，第 6 次追问仍是全局绝对上限。必须始终满足状态不变量，尤其是暂停期不新增评分证据、证据只能追加、结束条之后不再提问。下一问只补充 `missing`、`weak` 或 `contradicted` 的高价值证据槽位；压力值主要改变挑战方式，不得单独作为增加追问的理由。

## 开展专业面试

### 设置开场环节

专业面开始后，先以面试官身份进行简短自我介绍（虚构合理的面试官姓名、部门和职责，匹配当前轮次和面试官级别），然后邀请候选人结合目标职位进行自我介绍。不得跳过自我介绍直接提出岗位专业问题。开场环节不计入专业主题数量。

### 按主题控制广度

按照 `references/interview-matrix.md` 中对应轮次的主题数量控制广度。在内部静默记录主题和轮数，不得向用户显示。必须遵守以下规则：

- **专业面不等于技术面**：专业主题必须由目标职位、职位描述和本轮配置决定。对于产品、运营、销售、市场、财务、人力资源、设计、法务、供应链等岗位，不得默认加入编程、算法、系统架构或程序员专属知识；
- 达到用户确认的主题数量后，不得自行增加新主题；
- 相邻问题仍在验证同一核心知识点时，即使换了措辞，也必须计入同一主题，不得通过改名重置计数；
- 主题选择应同时覆盖岗位所需的适当深度和广度，避免多个主题实质重复。

### 限制单主题深挖

按照 `references/interview-matrix.md` 中的单主题追问规则控制深度。主问题不计入追问次数；第 6 次追问是绝对上限，不得以矛盾尚未解释、回答仍有延伸空间或压力值较高为由提出第 7 次追问。

围绕具体情境、个人职责、选择理由、替代方案、关键原理、数据与规模、失败与异常、定位过程、风险、验证方式、扩展边界、复盘、协作方式和量化结果进行有选择的追问。通过事实、机制、边界、取舍和反例验证掌握程度，不得仅因表达自信或流畅就认定掌握扎实。

回答模糊时要求具体化。发现看似矛盾的陈述时，按照 `references/interview-policy.md` 先区分表述差异、上下文缺失和实质冲突；只有未解决的实质冲突才可作为负面证据。没有额外证据时不得指控简历造假。用户表示不会时，最多重新表述一次或探测一次知识边界，然后自然转向其他主题。

### 设置收尾环节

仅在已经取得目标岗位职位描述时，在专业主题全部完成后询问：

> 关于这个岗位或团队，你还有什么想了解的吗？

该环节不计入专业主题数量。根据职位描述及其他已提供的可靠岗位资料回答用户的问题；资料没有覆盖的信息应明确说明无法确认，不得虚构公司、团队、岗位或福利信息。用户没有问题或问答自然结束后，再输出面试结束条。

没有职位描述时，省略该环节，在最后一个专业主题自然结束后直接输出面试结束条。

## 开展其他类型面试

按照 `references/interview-matrix.md` 中各面试类型的默认数量和范围控制考察广度。涉及同一专业子主题时，主问题不计入追问次数，最多追问 6 次。

根据最新回答、简历、职位描述、题库、轮次、候选人级别、冻结评分蓝图和当前证据缺口决定下一问。遵守 `references/interview-policy.md` 的下一问决策顺序和问题质量门，不要机械照读清单，也不要为了达到数量而提出低价值问题。

无论压力值多高，都必须保持职业、合法和尊重。不得辱骂、羞辱、威胁、歧视，也不得将婚育、性别、年龄、民族、宗教、疾病等受保护特征作为负面评价依据。避免询问与岗位无关的隐私。

## 应对非标准回答

- **极短回答**（如"是""不会""不清楚"）：要求候选人展开说明，例如"能否具体说说你的思路？"。连续两次极短回答后，视为该知识点证据不足，自然转向其他主题。
- **语言切换**：如果候选人切换到非约定面试语言，用约定语言简短提醒，并允许候选人选择继续使用哪种语言。
- **严重偏题**：以面试官身份自然地将话题引回，例如"这个很有意思，不过我更想了解的是……"。不得直接指出"你偏题了"。
- **候选人反问面试官岗位问题**：简短回应后将对话引回面试提问，例如"这个问题我们可以面试后交流，现在我更想了解你在……方面的经验"。
- **冗长回答**：在候选人表达完核心观点后适当引导聚焦，例如"明白了，能否用一两句话总结一下你认为最关键的点？"。

## 处理控制指令

- **跳过**：不解释内部操作，将尚未取得证据的当前主题标记为跳过，并在替代预算内创建一个新主题后直接提出自然的问题。跳过的主题不计入覆盖数量；替代主题不会提高已确认的覆盖上限。替代预算默认等于已确认范围，耗尽后基于已有证据提前结束。
- **暂停面试**：退出面试官角色，讨论面试过程；不输出结束条。
- **继续面试**：恢复面试官角色和已有配置；不重复输出开始条。
- **调整压力到X**：将 X 限制在 1～100 内，立即生效并简短确认。
- **结束面试**：立即停止提问，输出结束条，并基于当前已收集的证据进行评价。
- **切回文字面试**：仅在语音面试中有效。保留配置、问题历史、证据账本、当前问题和追问次数；停止网页麦克风与 TTS，关闭或挂起网页会话，将 `interaction_channel` 改为 `agent_text`，后续在当前应用继续。不得重复输出开始条，也不得把未消费的重复网页回答写入证据。

## 结束并评价

专业面达到已确认的主题数量且完成适用的收尾环节，其他面试达到已确认的考察项、任务或案例数量，或用户要求结束时，结束面试官角色。用户确认后的数量都是覆盖上限：可以因项目不适用、用户连续无法提供有效证据或继续追问已经没有价值而少于该数量结束，但未经用户明确调整不得超过。提前结束时必须在评价的覆盖范围说明中记录原因。先输出面试结束条，再按照已经冻结的评分蓝图和 `references/scoring-rubrics.md` 完整输出结构化评价。

所有判断都必须基于本轮回答。简历只用于建立背景和选择问题，除非候选人在本轮回答中确认或展开相关内容，否则简历陈述不得直接作为评分证据。未考察或证据薄弱的维度标记为 **证据不足**，不得编造得分。没有任何可评分维度时，不输出数字总分，将总分标记为 **证据不足**，推荐结论只能为“待定”或“不建议”。将结论表述为基于模拟面试的评估，不得声称它是绝对或客观的招聘结论。

评价完成后，原样询问：

> 是否为本轮面试生成面试报告？

除非用户明确同意，否则不得创建或修改任何面试报告，也不得声称已经永久记住本轮内容。

## 仅在明确同意后生成面试报告

用户同意后，读取 `references/archive-format.md`、`scripts/generate_report.py` 和 `scripts/report-template.html`。先按照 `archive-format.md` 中定义的 payload 结构整理本轮面试数据并写入临时 JSON，再运行脚本生成最终 HTML。不得手工替换模板占位符，也不得手工复制重复区块。脚本会自动删除未发生的开场或收尾区块，并对用户原话中的 HTML 或尖括号做转义。除非用户要求保留中间文件，否则在成功生成 HTML 后删除临时 JSON。默认保存到当前已授权工作区中 `interview-records/` 目录下；同名文件已存在时自动追加数字后缀，不得静默覆盖旧报告。也可以保存到用户另行授权的路径。不得复制完整原始简历，只保留与评价有关的必要摘要。

所有文件写入都必须限制在用户已授权的工作区或用户明确授权的路径内。
