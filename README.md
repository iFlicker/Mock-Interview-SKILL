# Mock-Interview

`Mock-Interview` 是一个基于真实简历开展沉浸式模拟面试的 Skill。它以真实面试官的方式进行回合制交流：每次只提出一个问题，根据候选人的回答自然追问，并在面试结束后依据本轮实际回答给出结构化评价。

## 核心特点

- **基于真实简历**：主动查找并读取用户提供的简历，不会在缺少简历时退化为通用题库面试。
- **沉浸式面试**：正式面试期间只呈现面试官会直接说的话，不显示题号、主题切换或内部进度。
- **主题驱动的专业面**：通过主题数量控制考察广度，不再用总问题数限制专业面。
- **控制深挖长度**：同一技术主题通常追问 3～5 轮，最多 6 轮，避免在单点上过度消耗。
- **适配不同候选人**：针对实习生、应届生和社招候选人调整考察重点。
- **覆盖多种面试形式**：支持人力资源面、专业面、主管面、高管面、行为面试、算法面试、系统设计、项目深挖、案例分析、压力面试和薪资谈判等。
- **基于证据评价**：只评价本轮实际考察到的能力；没有足够证据时明确标记为“证据不足”。
- **默认保护隐私**：未经用户明确同意，不创建面试记录，也不复制完整简历。

## 启动面试的必要条件

正式开始前必须依次满足以下条件：

1. **提供有效简历**  
   支持 PDF、DOC、DOCX、RTF、PNG、JPG、JPEG、WEBP、TXT 和 Markdown 等格式。简历无法读取、内容为空或图片无法识别时，需要提供可读取版本。

2. **明确目标职位**  
   目标职位必须由简历或用户明确给出，不会仅凭专业、项目经历或技能栈进行推断。

3. **确认面试轮次和类型**  
   例如专业一面、经理面、最终人力资源面、系统设计面试或薪资谈判。未说明时，Skill 会先提供相关选项和默认配置供用户确认。

职位描述、公司信息、预设题库、历史面试记录和指定考察范围不是启动面试的强制材料，但如果存在，Skill 会主动读取并纳入问题设计。

## 面试前的最终确认

Skill 会先展示紧凑的“本轮配置”，然后要求用户明确确认：

1. 使用 **纯模拟** 或 **教练模式**；
2. 是否还要调整重点方向、主题数量、压力值、面试官风格、语言或希望避开的话题。

默认反馈模式是纯模拟，但不会用默认值代替用户确认。只有用户明确确认反馈模式并完成自定义设置后，正式面试才会开始。

## 正式面试边界

配置确认后，会先输出：

```text
---------------------------- 面试开始 ----------------------------
```

正常结束或用户提前结束时，会先输出：

```text
---------------------------- 面试结束 ----------------------------
```

结束条之后才会出现总结与评价。暂停面试不会输出结束条，继续面试也不会重复输出开始条。

## 沉浸式输出规则

开始条与结束条之间不会出现：

- “第 1 题”“下一题”或其他题号；
- “切换主题”“进入下一个主题”；
- 已完成题数、主题数或剩余进度；
- “这是一个追问”等内部提问说明；
- 评分意图或后续提问计划。

主题变化通过自然的面试问题完成。纯模拟模式下只呈现真实面试官会直接说的话。

## 支持的候选人和面试类型

### 候选人类型

- 实习生
- 应届生
- 社招候选人
- 高级岗位或管理岗位候选人

### 面试类型

- 人力资源初筛和最终人力资源面
- 专业一面至专业五面
- 资深员工、组长、经理、总监和高管面
- 行为面试
- 编程或算法面试
- 系统设计面试
- 项目深挖和作品答辩
- 案例或业务分析面试
- 跨部门面试
- 压力面试
- 薪资谈判演练

## 专业面主题数量

专业面不设置总问题数上限，而是限制实质不同的考察主题数量：

| 专业轮次 | 默认主题数 | 可调整范围 |
|---|---:|---:|
| 专业一面 | 5 | 4～7 |
| 专业二面／资深员工面 | 6 | 5～7 |
| 专业三面／经理面 | 6 | 5～7 |
| 专业四面／总监面 | 5 | 4～6 |
| 专业五面／高管面 | 4 | 3～5 |

主题可以是项目经历、专业基础、系统设计、问题排查、方案取舍、业务结果、跨团队协作或管理决策等。开场自我介绍和基于职位描述的收尾问答不计入主题数量。

### 单主题深挖

- 通常围绕一个技术主题进行 3～5 轮。
- 主问题和同主题追问都计入轮数。
- 最多进行 6 轮；第 6 轮回答完成后必须转向其他主题。
- 用户明确不会、证据充分或继续追问没有价值时，可以提前结束该主题。
- 轮数只在内部记录，不会展示给候选人。

## 专业面的开场和收尾

### 开场

专业面不会直接从技术问题开始。面试开始后的第一个问题会邀请候选人结合目标职位进行自我介绍。该环节不计入专业主题数量。

### 收尾

如果已经提供目标岗位职位描述，专业主题完成后会询问：

> 关于这个岗位或团队，你还有什么想了解的吗？

回答只依据职位描述及其他可靠岗位资料；资料没有覆盖的信息会明确说明无法确认，不会虚构。

如果没有职位描述，则省略该环节，在最后一个专业主题结束后直接结束面试。

## 其他面试的范围控制

- 人力资源初筛：默认 5 个考察项，合理范围 3～10。
- 最终人力资源面：默认 8 个考察项，合理范围 3～10。
- 编程或算法面试：默认 3 个独立任务，可调整为 2～4。
- 系统设计面试：默认 1 个完整案例，可调整为 1～2。
- 项目深挖：默认 3 个项目或实质不同的项目主题，可调整为 2～4。
- 作品集或方案答辩：默认 2 个作品或方案，可调整为 1～3。
- 案例或业务分析：默认 2 个独立案例，可调整为 1～3。

这些面试中涉及同一技术子主题的连续问题，同样最多追问 6 轮。

## 本轮配置

开始前展示的配置通常包含：

| 配置项 | 说明 |
|---|---|
| 候选人类型 | 实习生、应届生或社招候选人 |
| 面试轮次与类型 | 当前轮次、面试形式和面试官级别 |
| 范围控制 | 专业面的主题数，或其他面试的考察项、任务与案例数量 |
| 面试官风格 | 友善、严谨、直接、结果导向等 |
| 压力值 | 1～100，可在面试过程中即时调整 |
| 面试语言 | 默认跟随用户当前使用的语言 |
| 重点方向 | 根据目标职位、职位描述、简历和轮次确定 |
| 专项环节 | 是否包含编程、案例分析或系统设计 |
| 反馈模式 | 必须明确确认纯模拟或教练模式 |
| 题库使用 | 是否参考用户提供或工作区中发现的题库 |
| 避免话题 | 用户希望避开的敏感话题 |

## 反馈模式

### 纯模拟

面试过程中不即时点评，也不提前泄露参考答案；所有反馈在面试结束后统一提供。

### 教练模式

每次回答后提供简短反馈，再继续面试。即使在教练模式下，也不会暴露内部主题计数、提问策略或评分进度。

## 面试中的控制指令

用户可以随时使用：

- **跳过**：跳过当前问题，并自然进入其他问题。
- **暂停面试**：退出面试官角色，讨论面试过程。
- **继续面试**：恢复面试。
- **调整压力到X**：将压力值调整到 1～100 范围内的指定数值。
- **结束面试**：立即停止提问，输出结束条，并基于当前已有证据进行评价。

## 评价内容

面试结束后会提供：

1. 本轮面试概览和实际主题或考察项覆盖情况
2. 0～100 的总分及评分覆盖度
3. 动态加权的分维度评分
4. 有具体回答证据支持的优势
5. 主要风险和不足
6. 回答中的矛盾、模糊点或知识缺口
7. 最值得改进的 3～5 项内容
8. 关键问题的更优回答思路
9. 是否建议进入下一轮
10. 下一轮可能重点考察的方向

用户提前结束面试时，仍会基于已有回答进行评价；没有考察到的部分会标记为“证据不足”。

## 留档规则

评价完成后，Skill 会询问：

> 是否将本轮面试留档？默认不留档。

只有用户明确同意后才会创建记录。默认记录保存到当前已授权工作目录下的 `interview-records/`。留档流程不再手工拼接 HTML，而是先整理结构化 JSON，再调用 `scripts/generate_report.py` 生成最终 HTML 报告。

生成脚本会自动展开重复区块、删除未发生的开场或收尾环节，并对用户原话做 HTML 转义。留档不会复制完整原始简历，只保留支持评价所必需的摘要。所有文件写入都限制在用户已授权的工作区或明确授权的路径中。

## 使用示例

直接调用：

```text
使用 $mock-interview，根据我的简历进行一场后端开发专业二面。
```

指定配置：

```text
使用 $mock-interview 模拟高级产品经理的经理面，7 个主题，压力值 70，纯模拟模式，重点考察跨部门协作和业务取舍。
```

薪资谈判：

```text
使用 $mock-interview 进行最终人力资源面和薪资谈判演练，目标职位是算法工程师。
```

如果简历已经位于工作目录中，也可以直接说明文件位置：

```text
使用 $mock-interview，读取 ./求职材料/简历.pdf，模拟系统设计面试。
```

## 文件结构

```text
mock-interview/
├── README.md
├── SKILL.md
├── agents/
│   └── openai.yaml
├── scripts/
│   ├── generate_report.py
│   ├── report-template.html
│   └── test_generate_report.py
└── references/
    ├── archive-format.md
    ├── interview-matrix.md
    └── scoring-rubrics.md
```

- `SKILL.md`：核心工作流程和强制行为规则。
- `agents/openai.yaml`：Skill 的界面显示信息和默认调用提示。
- `scripts/generate_report.py`：根据结构化 JSON 生成最终 HTML 报告。
- `scripts/report-template.html`：由生成脚本消费的 HTML 模板。
- `scripts/test_generate_report.py`：覆盖模板展开、可选区块删除和 HTML 转义的回归测试。
- `references/interview-matrix.md`：候选人类型、轮次策略、主题数量、默认配置和专项面试规则。
- `references/scoring-rubrics.md`：动态评分方法和最终评价结构。
- `references/archive-format.md`：用户同意留档后的保存规则、JSON payload 结构和脚本调用方式。
